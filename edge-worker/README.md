# Leviathan Edge Gateway

Optional Cloudflare Worker designed to place a **lightweight, fast, and controlled gateway** in front of Leviathan.

The idea is simple: the VPS remains the core of Leviathan, while the Edge Gateway sits in front as an intelligent first layer for lightweight requests, cache, prewarm, and origin protection.

## Why use it

Leviathan can work without the Edge Gateway, but placing the Worker in front adds an extra layer:

- faster responses for manifest, configure, catalog, meta, subtitles, and assets;
- less pressure on the VPS for repetitive requests;
- stale fallback to keep previously seen lightweight responses available during micro-outages or restarts;
- best-effort prewarm when a stream request arrives;
- secret header toward the origin, useful for distinguishing edge traffic from direct traffic;
- direct redirect to the origin for HLS/video proxy traffic, so the Worker is not wasted on heavy segments.

## Public Leviathan instance

The **public Leviathan instance already benefits from this architecture**: edge cache, accelerated lightweight requests, prewarm hints, and cautious fallback are designed to work together without changing the normal installation flow.

In practice, users of the public instance already get the gateway benefits when traffic passes through the configured Edge Gateway, while Leviathan continues to behave like Leviathan: stable, direct, modular, and without fragile magic layered on top of video streams.

## What it does

- forwards Stremio requests to the Leviathan VPS;
- applies lightweight cache for manifest, configure, catalog, meta, subtitles, and assets;
- adds a secret header toward the origin;
- sends best-effort hints to `/internal/edge/prewarm` when it sees a stream request;
- uses stale fallback for previously seen lightweight responses;
- avoids proxying HLS/video segments: by default, it redirects them directly to the origin.

## What it does NOT do

- it does not replace FlareSolverr;
- it does not solve Cloudflare challenges from providers;
- it does not scrape;
- it should not proxy heavy `.ts`, `.m4s`, or `.m3u8` segments;
- it does not change `public/index.html` or `public/smartphone.js`;
- it does not force the installation URL: it remains the one generated by the existing code.

## Quick deploy

```bash
cd edge-worker
cp wrangler.toml.example wrangler.toml
# edit ORIGIN_URL and EDGE_SECRET
npx wrangler deploy
```

In Leviathan's `.env`, set the same secret:

```env
LEVIATHAN_EDGE_ENABLED=true
LEVIATHAN_EDGE_SECRET=change-this-long-secret
LEVIATHAN_EDGE_ALLOW_DIRECT=true
LEVIATHAN_EDGE_REQUIRE_SECRET=false
LEVIATHAN_EDGE_INTERNAL_ENABLED=true
```

When you want to force public traffic through the Edge Gateway:

```env
LEVIATHAN_EDGE_ALLOW_DIRECT=false
LEVIATHAN_EDGE_REQUIRE_SECRET=true
```

Do this only after verifying that the Worker URL works.

## Recommended mode for Leviathan

Use the Worker as an **accelerator/gateway**, not as a universal video proxy.

Safe configuration:

```env
LEVIATHAN_EDGE_ALLOW_DIRECT=true
LEVIATHAN_EDGE_REQUIRE_SECRET=false
LEVIATHAN_EDGE_CACHE_STREAM_SECONDS=0
```

This mode lets Leviathan keep responding directly, while enabling edge benefits whenever traffic passes through the Worker.

If you want to close the origin and accept only Worker-signed traffic, switch to the stricter mode:

```env
LEVIATHAN_EDGE_ALLOW_DIRECT=false
LEVIATHAN_EDGE_REQUIRE_SECRET=true
```

Use it only after thoroughly testing the Worker, because a wrong `EDGE_SECRET` or an incorrect origin URL can cut off public traffic.

## Stream metadata micro-cache

By default, stream metadata is not cached:

```env
LEVIATHAN_EDGE_CACHE_STREAM_SECONDS=0
```

If you want an aggressive but still cautious micro-cache, you can use:

```env
LEVIATHAN_EDGE_CACHE_STREAM_SECONDS=10
# or
LEVIATHAN_EDGE_CACHE_STREAM_SECONDS=20
```

Do not set this too high if you use personal configurations, tokens, or sensitive parameters in the path.

## Golden rule

The Worker should make Leviathan faster and more resilient, not more complicated.

- lightweight requests: cache, stale fallback, prewarm;
- video/HLS requests: direct redirect to the origin;
- security: secret header and strict mode only when you are sure;
- public UX: the public instance already benefits from the gateway without changing the expected user behavior.
